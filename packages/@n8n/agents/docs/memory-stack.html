Base memory

Agent memory stack

Base Memory gives an agent two model-facing memory sources: a user profile for durable preferences and context, and session memory for the current thread objective.

Enabled by memory.enabled: true in n8n agent config.
Includes <user-profile> and <session-memory>.
Does not include Case memory, embeddings, or recall_memory.
Profile scope scopeKind + agentId + resourceId.

Prompt blocks

The runtime injects memory as explicit XML-like blocks before the LLM call. Empty blocks are omitted.

<memory_blocks>
<user-profile>
- Prefers concise support-debugging answers.
- Uses n8n for customer-support automation.
</user-profile>

<session-memory>
# Thread memory

## Current objective
Debug the Priority Escalation workflow.
</session-memory>
</memory_blocks>

<memory> is reserved for Case memory. It appears only when episodic memory is enabled in the downstream Case memory PR.

User profile

The user profile is a concise markdown bullet list of durable information this agent should remember about this user or resource.

What belongs here
  • Stable user preferences and communication style.
  • Durable environment, tooling, or domain context.
  • Guidance that should personalize future answers for this agent.
What stays out
  • Current task state, debugging steps, and next actions.
  • Temporary workflow names or throwaway scenarios.
  • Agent instructions or a summary of the agent description.

Session memory

Session memory is thread-scoped working memory. It tracks the active objective and current progress, not durable user profile facts.

Captured by default
  • Current objective and task state.
  • Objective-specific decisions and constraints.
  • Open follow-ups and uncertainties for the active thread.
Updated by
  • The observer path after successful turns.
  • The compactor path when enough observations have accumulated.
  • The configured memory backend through thread metadata.

Storage and UI

  • agents_memory_profiles stores user profiles by scopeKind, agentId, and resourceId.
  • The Agents Memory panel shows a read-only User profile section using the same compact editor style as session memory.
  • n8n JSON config does not expose profile settings; profiles are implicit when base Memory is enabled.
  • The SDK still exposes Memory.profiles(...) explicitly for SDK consumers.
