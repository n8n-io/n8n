Instance AI → Agents SDK

Ideas for moving generic agent infrastructure out of @n8n/instance-ai and into @n8n/agents.

Use ← → or click the sides of the screen. Companion deck: packages/cli/src/modules/agents/docs/workshop-agents-module-to-sdk.html

Context

Two layers today

@n8n/agents

Fluent agent builder, runtime loop, memory, MCP, workspace, sub-agent primitives, telemetry hooks.

@n8n/instance-ai

Product agent: orchestrator factory, domain tools, planning, SSE events, tracing glue, n8n adapters.

Workshop question: Which instance-ai code is really “agent platform” vs “n8n product”?

#1 · High overlap

Custom delegate tool vs SDK sub-agent

Instance AI ships delegate.tool.ts + sub-agent-factory.ts. The SDK already has createDelegateSubAgentTool() with inline child runs, tool blocklists, and lifecycle events.

SDK candidate: Host hooks for briefing/debriefing, detached runs, and trace attribution — so products plug in policy instead of reimplementing delegation.

#2 · Parallel concepts

Planned tasks vs write_todos

Instance AI has a full task graph (create-tasks, dependency validation, user approval, checkpoint kinds). The SDK exposes a simpler in-run todo list via write_todos.

SDK candidate: Optional persisted task-graph primitive (validate deps, schedule, status) — instance-ai would keep n8n-specific task kinds like build-workflow.

#3 · Stream plumbing

Resumable stream + HITL loop

resumable-stream-executor.ts and consume-with-hitl.ts wrap SDK streams: consume chunks, detect suspension, wait for confirmation, resume, repeat.

SDK candidate: A first-class consumeStream({ onSuspend, resume }) helper. Products map confirmations to their own UI; the suspend/resume cycle stays generic.

#4 · MCP hardening

Tool name validation & schema sanitization

Instance AI validates MCP names, resolves collisions with domain tools, and walks Zod trees to fix Anthropic-incompatible schemas (mcp-tool-name-validation.ts, sanitize-mcp-schemas.ts).

SDK candidate: Built into McpClient / tool resolution — safe names, schema normalization, skip-with-warning policy. Every MCP consumer benefits.

#5 · Lifecycle

Background task manager

Instance AI tracks detached sub-agents with dedupe keys, liveness timeouts, mid-flight corrections, and planned-task linkage. SDK only has a lightweight in-flight promise tracker.

SDK candidate: Managed background-run registry (spawn, cancel, correct, timeout policy). Instance AI adds n8n-specific dedupe and checkpoint semantics on top.

#6 · State

Run state & suspension registry

run-state-registry.ts holds active runs, suspended HITL state, confirmation payloads, and checkpoint metadata across HTTP requests.

SDK candidate: Serializable run-state store interface + default in-memory impl. CLI/redis persistence stays in the host, but the shape and transitions live in the SDK.

#7 · Tool surface

Tool registry & deferred splitting

tool-registry.ts merges/filters tool maps. instance-agent.ts splits core vs deferred tools with product-specific “always loaded” sets.

SDK candidate: Tool-set utilities and a declarative deferred-tool policy (pin list + search config) instead of ad-hoc Maps in each product.

#8 · Events

Stream chunk → UI event mapping

map-chunk.ts translates SDK StreamChunk into Instance AI SSE events (text deltas, tool calls, domain-specific payloads).

SDK candidate: Canonical product-neutral event types + mapper from runtime chunks. Hosts extend with domain events; transport (SSE) stays outside the SDK.

#9 · Workspace

Scoped workspace jail

scoped-workspace.ts wraps filesystem access so builder agents cannot escape their sandbox root.

SDK candidate: workspace.scope(root) as a first-class API on the existing Workspace package — reusable for any sandboxed agent.

#10 · Observability

Trace input assembly

langsmith-tracing.ts (~1.8k lines) builds trace payloads from prompts, tools, skills, wraps tools, and coordinates replay — mostly around SDK telemetry.

SDK candidate: Standard trace snapshot builder (model, tools, memory config, skills). Product layer keeps proxy auth, replay, and LangSmith project naming.

#11 · Memory

Sub-agent persistence pattern

agent-persistence.ts creates hidden memory threads for sub-agents so HITL suspend/resume survives across requests.

SDK candidate: Ephemeral child-run persistence helper (resource IDs, parent linkage metadata). Host supplies the storage backend.

#12 · Aggregation

Work summary during streams

work-summary-accumulator.ts collects tool outcomes while the stream runs — used for debriefing and orchestrator follow-ups.

SDK candidate: Optional stream sidecar that aggregates tool results/errors into a structured summary object at run end.

#13 · Reliability

Liveness & timeout policy

liveness-policy.ts defines idle/stuck detection for long background builder runs with configurable surfaces and reasons.

SDK candidate: Generic activity watchdog hook on background runs (last activity timestamp + policy callbacks).

#14 · MCP hosting

MCP client manager wrapper

Instance AI caches MCP connections by config hash, applies SSRF gates, sanitizes schemas, and merges local gateway + external servers.

SDK candidate: Cached McpClient factory with pluggable URL validators and post-process hooks — less boilerplate in every host.

#15 · Mostly done

Output redaction wrapper

output-redaction.ts applies SDK StreamingRedactor to product events before SSE. Thin layer; SDK already owns the redactor.

SDK candidate: Optional event-bus middleware in SDK — instance-ai would only supply default categories and logging.

Stays in instance-ai

Not SDK material

  • Domain tools — workflows, executions, credentials, nodes, data tables
  • Workflow builder & loop — TS SDK parsing, verify/debug state machine
  • System prompts & skills content — n8n-specific orchestration guidance
  • SSE transport & thread storage — HTTP/API contract with editor-ui
  • CLI adapter layer — RBAC, n8n services, credits, gateway daemon
