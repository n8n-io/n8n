# @n8n/agents SDK Design

**Date:** 2026-02-25
**Status:** Draft
**Scope:** Design for a new TypeScript agent SDK that provides agent primitives for n8n's code-first execution engine

---

## Context

n8n is building a new code-first workflow engine as a parallel offering to the
existing visual canvas editor. A separate team handles the execution runtime
(durable execution, checkpointing, resume). This document designs the **agent
SDK** — the package developers (and AI coding agents like Claude Code) use to
define and run agents in TypeScript.

The existing canvas-based agent system (LangChain-powered, EngineRequest/
EngineResponse protocol) is irrelevant to this design. This is greenfield.

## Requirements

- **TypeScript-first** — full type safety, Zod schemas throughout
- **Provider-neutral** — must support any major LLM provider
- **Framework-neutral** — must be swappable underneath without changing the user-facing API
- **AI-consumable** — primarily generated by AI coding agents; minimal API surface, builder pattern, self-documenting
- **Full stack from day one** — agents, tools, memory, multi-agent, evals, guardrails, observability
- **Engine-cooperative** — the SDK runs the agent loop but cooperates with the execution engine for durability, checkpointing, and resource limits
- **Tools are TypeScript functions** — plain async functions with Zod schemas, no n8n node system integration needed initially

## Architectural Decision: Mastra as Runtime Backend

After evaluating Vercel AI SDK, Mastra, and a home-rolled approach, the
decision is to use **Mastra as the runtime backend behind an n8n-owned
abstraction layer**.

```
User code (generated by Claude Code)
        | uses
@n8n/agents  <-- stable API surface (n8n owns this contract)
        | delegates to
@mastra/core + @mastra/memory + @mastra/evals  <-- runtime (swappable)
        | uses
Vercel AI SDK  <-- provider layer (Mastra uses this internally)
        ^ cooperates with
n8n execution engine  <-- durability, checkpointing, resource limits
```

### Why Mastra

- **Full runtime from day one.** Memory (three-tier), evals (12+ scorers),
  guardrails (PII, injection, moderation), observability (auto-tracing) — all
  production-ready. Building these on AI SDK primitives alone would mean
  rebuilding what Mastra already built.
- **Apache 2.0 licensed.** No licensing concerns.
- **Built on Vercel AI SDK.** Uses AI SDK's provider system internally, so
  provider neutrality comes for free (25+ official providers).
- **Active maintenance.** YC W25, ex-Gatsby team, 21.3k stars, 437k npm/week.

### Why Not AI SDK Alone

AI SDK provides excellent primitives (`generateText`, `ToolLoopAgent`,
provider system) but requires building memory, evals, guardrails, multi-agent
orchestration, and observability from scratch. That is effectively rebuilding
Mastra.

### Why Not Home-Roll

The provider abstraction layer alone (25+ providers, streaming, tool calling,
structured output edge cases) took Vercel's team years. Not realistic for full
stack from day one.

### Lock-in Mitigation

Nothing from `@mastra/*` is ever exposed to user code. The `@n8n/agents`
package is the only import consumers use. If Mastra needs to be replaced, the
internals of `@n8n/agents` are rewritten — user code is unaffected.

### What We Skip from Mastra

Mastra's workflow engine is not used. It overlaps with the execution runtime
team's scope. We use only: agent runtime, memory, evals, guardrails, and
observability.

## Package Structure

```
packages/@n8n/agents/
  src/
    index.ts                 # Public API — the only thing consumers import
    agent.ts                 # Agent builder
    tool.ts                  # Tool builder
    memory.ts                # Memory builder
    guardrail.ts             # Guardrail builder
    scorer.ts                # Eval scorer builder
    network.ts               # Multi-agent network builder
    run.ts                   # Run object (execution handle + events)
    types.ts                 # All public TypeScript types
    runtime/                 # Internal — never exported
      mastra-adapter.ts      # Translates n8n types -> Mastra calls
      engine-bridge.ts       # Cooperation interface with n8n engine
      provider-registry.ts   # Model string -> AI SDK provider mapping
  package.json
  tsconfig.json
```

Everything in `src/runtime/` is an implementation detail. The public API is:

```typescript
import { Agent, Tool, Memory, Guardrail, Scorer, Network } from '@n8n/agents';
```

## Core API

The SDK uses a **builder-only pattern**. One way to create each primitive.
No config-object alternatives. The builder guides AI code generators through
valid construction paths via chaining.

### Tool

```typescript
import { Tool } from '@n8n/agents';
import { z } from 'zod';

const searchTool = new Tool('web-search')
  .description('Search the web for information')
  .input(z.object({
    query: z.string().describe('Search query'),
    maxResults: z.number().default(5),
  }))
  .output(z.object({
    results: z.array(z.object({
      title: z.string(),
      url: z.string(),
      snippet: z.string(),
    })),
  }))
  .handler(async ({ query, maxResults }) => {
    const results = await fetch(`https://api.search.com?q=${query}&n=${maxResults}`);
    return { results: await results.json() };
  })
  .build();
```

**Design choices:**
- `.input()` and `.output()` use Zod — same as Mastra and AI SDK, zero translation
- `.handler()` receives validated input directly (not wrapped in `{ inputData }`)
- `.description()` is explicit in the chain — reinforces that it matters for LLM tool selection
- `.output()` is optional but recommended — enables runtime validation

#### Tool Approval

```typescript
const deleteTool = new Tool('delete-record')
  .description('Delete a database record')
  .input(z.object({ id: z.string() }))
  .requiresApproval()
  .handler(async ({ id }) => { /* ... */ })
  .build();

// Conditional approval
const paymentTool = new Tool('process-payment')
  .description('Process a payment')
  .input(z.object({ amount: z.number() }))
  .requiresApproval(({ amount }) => amount > 1000)
  .handler(async ({ amount }) => { /* ... */ })
  .build();
```

#### Tool Pause (Request Human Input)

```typescript
const researchTool = new Tool('deep-research')
  .description('Research that may need human guidance')
  .input(z.object({ query: z.string() }))
  .handler(async ({ query }, ctx) => {
    const results = await search(query);
    if (results.length === 0) {
      await ctx.pause({
        reason: 'No results found',
        requestedInput: z.object({ alternativeQuery: z.string() }),
      });
    }
    return { results };
  })
  .build();
```

### Agent

```typescript
import { Agent } from '@n8n/agents';

const agent = new Agent('researcher')
  .model('anthropic/claude-sonnet-4')
  .instructions('You are a research assistant. Be concise.')
  .tool(searchTool)
  .tool(saveTool)
  .build();

// Execute
const result = await agent.run('Find recent papers on RAG architectures');
console.log(result.text);

// Stream
const stream = await agent.stream('Find recent papers on RAG architectures');
for await (const chunk of stream) {
  process.stdout.write(chunk.text);
}
```

The builder enforces required fields at `.build()` time — calling `.build()`
without `.model()` throws a clear error.

### Memory

Memory is a separate builder, attached to agents via `.memory()`:

```typescript
import { Agent, Memory } from '@n8n/agents';

// Simple — conversation history with a window
const agent = new Agent('assistant')
  .model('anthropic/claude-sonnet-4')
  .instructions('You are a helpful assistant.')
  .memory(
    new Memory()
      .lastMessages(20)
      .build()
  )
  .build();

// Full-featured — history + semantic recall + working memory
const agent = new Agent('assistant')
  .model('anthropic/claude-sonnet-4')
  .instructions('You are a personal assistant.')
  .memory(
    new Memory()
      .lastMessages(20)
      .semanticRecall({ topK: 4, messageRange: { before: 1, after: 1 } })
      .workingMemory(
        z.object({
          name: z.string().optional(),
          preferences: z.record(z.string()).optional(),
        })
      )
      .build()
  )
  .build();
```

**Runtime context** — `resourceId` and `threadId` are provided at execution
time, not build time, since the same agent definition serves many users:

```typescript
await agent.run('My name is Alice', {
  resourceId: 'user-123',
  threadId: 'conversation-abc',
});
```

Storage backend is not exposed. Behind the facade, Mastra handles persistence.
The engine cooperation interface will eventually let the engine provide storage.

### Multi-Agent

Two patterns: **agent-as-tool** (explicit delegation) and **networks**
(coordinated routing).

#### Agent as Tool

```typescript
const researcher = new Agent('researcher')
  .model('anthropic/claude-sonnet-4')
  .instructions('You research topics thoroughly.')
  .tool(searchTool)
  .build();

const writer = new Agent('writer')
  .model('anthropic/claude-sonnet-4')
  .instructions('You write clear, engaging content.')
  .build();

const orchestrator = new Agent('orchestrator')
  .model('anthropic/claude-sonnet-4')
  .instructions('Coordinate research and writing tasks.')
  .tool(researcher.asTool('Delegate research tasks'))
  .tool(writer.asTool('Delegate writing tasks'))
  .build();

await orchestrator.run('Research RAG architectures and write a summary');
```

`.asTool(description)` converts any built agent into a tool. The
orchestrator's LLM decides when to delegate.

#### Agent Network

```typescript
import { Network } from '@n8n/agents';

const network = new Network('content-team')
  .coordinator(
    new Agent('coordinator')
      .model('anthropic/claude-sonnet-4')
      .instructions('Route tasks to the right specialist.')
      .build()
  )
  .agent(researcher)
  .agent(writer)
  .agent(editor)
  .build();

const result = await network.run(
  'Research top cities in Japan, write a travel guide, then edit for tone'
);
```

**Design choices:**
- Agent-as-tool is the primary pattern — explicit, debuggable, easy for AI to generate
- Network is a convenience for LLM-routed multi-agent coordination
- Both compose — a network can contain agents that use agent-as-tool internally
- The coordinator is an explicit agent with its own model and instructions (no implicit magic)

### Guardrails

Input/output processors declared on the agent, executed automatically by the
runtime:

```typescript
import { Agent, Guardrail } from '@n8n/agents';

const agent = new Agent('secure-assistant')
  .model('anthropic/claude-sonnet-4')
  .instructions('You are a helpful assistant.')
  .inputGuardrail(
    new Guardrail('pii-detector')
      .type('pii')
      .strategy('redact')
      .detect(['email', 'phone', 'credit-card'])
      .build()
  )
  .inputGuardrail(
    new Guardrail('injection-detector')
      .type('prompt-injection')
      .strategy('block')
      .threshold(0.8)
      .build()
  )
  .outputGuardrail(
    new Guardrail('moderation')
      .type('moderation')
      .strategy('block')
      .build()
  )
  .build();
```

**Strategies:**
- `block` — reject the input/output, Run emits `stateChange` to `blocked`
- `redact` — sanitize and continue
- `warn` — flag but continue

Built-in types (`pii`, `prompt-injection`, `moderation`) map to Mastra's
processors behind the facade.

### Evals

Scorers attach to agents and run automatically with configurable sampling:

```typescript
import { Agent, Scorer } from '@n8n/agents';

const agent = new Agent('evaluated-assistant')
  .model('anthropic/claude-sonnet-4')
  .instructions('You are a helpful assistant.')
  .scorer(
    new Scorer('relevancy')
      .type('answer-relevancy')
      .model('anthropic/claude-haiku-4-5')
      .sampling(0.5)
      .build()
  )
  .scorer(
    new Scorer('toxicity')
      .type('toxicity')
      .model('anthropic/claude-haiku-4-5')
      .sampling(1.0)
      .build()
  )
  .build();
```

Eval results flow through the event system — the Run emits `eval` events that
the engine can collect and surface.

### Observability

Tracing is automatic and baked into the runtime. The user never opts in.
The engine configures where traces go:

```typescript
// Engine code, not user code
import { configure } from '@n8n/agents';

configure({
  tracing: {
    exporter: langfuseExporter,
  },
});
```

Every Run automatically produces traces: LLM calls, tool executions, memory
operations, guardrail checks, eval scores.

## Run Object: Engine Cooperation Interface

`agent.run()` returns a `Run` object — the bridge between the agent runtime
and the execution engine. The user never interacts with Run directly; it is
the engine's API into the agent.

```
agent.build()  ->  Agent        (the definition — user's domain)
agent.run()    ->  Run          (the execution — engine's domain)
                     .on()          observe
                     .approve()     control
                     .deny()        control
                     .resume()      control
                     .abort()       control
                     .result        await final output
```

### Events (Automatic — Emitted by the Runtime)

```typescript
const run = agent.run('Delete record 123', {
  resourceId: 'user-123',
  threadId: 'conv-abc',
});

run.on('step', ({ step, toolCalls, tokens }) => { });
run.on('toolCall', ({ tool, input, output, duration }) => { });
run.on('message', ({ role, content }) => { });
run.on('stateChange', ({ from, to, context }) => { });
run.on('eval', ({ scorer, score, reasoning }) => { });
run.on('error', ({ error, step, recoverable }) => { });
```

### State Machine

```
running -> waiting_approval -> running (approved) or failed (denied)
running -> paused -> running (resumed) or failed (aborted)
running -> blocked (guardrail) -> failed
running -> completed
running -> failed
```

### Control (Engine Responds to Signals)

```typescript
run.on('stateChange', async ({ to, context }) => {
  if (to === 'waiting_approval') {
    await run.approve(context.approvalId);
    // or: await run.deny(context.approvalId, 'Not authorized');
  }
  if (to === 'paused') {
    await run.resume(context.pauseId, { alternativeQuery: 'try this' });
  }
});

const result = await run.result;
```

The engine does not need to understand agent internals. It subscribes to
state changes and responds with approve/deny/resume/abort. The framework
handles everything else.

## Complete Example

```typescript
import { Agent, Tool, Memory, Guardrail, Scorer, Network } from '@n8n/agents';
import { z } from 'zod';

// --- Tools ---

const searchTool = new Tool('search')
  .description('Search the web')
  .input(z.object({ query: z.string() }))
  .handler(async ({ query }) => ({ results: await search(query) }))
  .build();

const writeTool = new Tool('write-file')
  .description('Write content to a file')
  .input(z.object({ path: z.string(), content: z.string() }))
  .requiresApproval()
  .handler(async ({ path, content }) => {
    await writeFile(path, content);
    return { written: true };
  })
  .build();

// --- Agents ---

const researcher = new Agent('researcher')
  .model('anthropic/claude-sonnet-4')
  .instructions('You research topics thoroughly and return structured findings.')
  .tool(searchTool)
  .memory(
    new Memory()
      .lastMessages(20)
      .semanticRecall({ topK: 4 })
      .build()
  )
  .inputGuardrail(
    new Guardrail('injection')
      .type('prompt-injection')
      .strategy('block')
      .build()
  )
  .scorer(
    new Scorer('relevancy')
      .type('answer-relevancy')
      .model('anthropic/claude-haiku-4-5')
      .sampling(0.5)
      .build()
  )
  .build();

const writer = new Agent('writer')
  .model('anthropic/claude-sonnet-4')
  .instructions('You write clear, engaging content based on research.')
  .tool(writeTool)
  .build();

// --- Multi-Agent ---

const team = new Network('content-team')
  .coordinator(
    new Agent('coordinator')
      .model('anthropic/claude-sonnet-4')
      .instructions('Route tasks to researcher or writer as appropriate.')
      .build()
  )
  .agent(researcher)
  .agent(writer)
  .build();

// --- Execute ---

const run = team.run(
  'Research RAG architectures and write a summary document',
  { resourceId: 'user-123', threadId: 'project-abc' },
);

const result = await run.result;
```

## Open Questions

1. **Memory storage backend.** Currently hidden behind the facade. When the
   engine provides storage, how does the memory builder expose backend
   selection? Likely an engine-level configuration, not user-facing.

2. **Model fallback arrays.** Mastra supports `model: [primary, fallback1, fallback2]`.
   Should the Agent builder expose `.fallbackModel()` or is that engine-level config?

3. **Custom guardrails.** The built-in types cover common cases. How do users
   define custom guardrail logic? Likely a `.handler()` on the Guardrail builder,
   similar to Tool.

4. **Custom scorers.** Same pattern — need a way to define scoring logic beyond
   the built-in types.

5. **Streaming semantics.** `agent.stream()` returns an async iterable for
   token streaming. How does this interact with the Run event system? The Run
   object likely wraps the stream and emits events alongside token delivery.

6. **Engine bridge interface.** The exact contract between the Run object and
   the execution engine (checkpointing, resource limits, credential access)
   needs to be co-designed with the runtime team.
