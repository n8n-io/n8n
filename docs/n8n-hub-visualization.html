n8n hub · live walkthrough

Make every n8n node callable from outside a workflow.

A hackathon design for an HTTP API, an MCP server, and a TypeScript SDK that share one primitive — execute one node with a credential. This page walks through the full interaction live, from prompt to execution to UI.

Filipe Tavares · 2026-05-10 · Status: hackathon scoped, not started

02What we add: three meta-tools

If we naively exposed every (node × resource × operation) tuple we'd register thousands of tools — Anthropic's guidance is to keep it under ~30 active tools per session. So we don't expose the catalog directly. We expose three tools that search it.

ToolPurposeReturns
n8n_search_tools Find tools by intent. Filterable by category, by credential availability, by node name. Backed by the existing NodeCatalogService — semantic search across every loaded node. Ranked list of (toolId, name, description, inputSchema, userCredentials)
n8n_execute_tool Run a tool. Validates params against schema, dispatches to POST /executions/node. Supports dryRun: true for risky operations. { executionId, status, output } — or { status: 'dry_run', wouldExecute }
n8n_list_credentials List the user's connected credentials, optionally filtered by node type. Available for the disambiguation case — common path skips this since search returns credentials inline. [{ id, name, type }] — never includes secret material
The credential model The MCP server never returns secret material. Clients pass credentialId; n8n resolves the credential server-side at execution time. A leaked token can call nodes — it cannot extract the underlying Slack/Postgres/Gmail token.

Dry-run mode

For high-impact actions, an agent can verify what would happen before committing. Cheap insurance, pairs with HITL approval later.

// Request
{
  "id": "slack.message.send",
  "credentialId": "cred_3a8f",
  "dryRun": true,
  "params": { "channel": "#all-company", "text": "Wide announcement" }
}

// Response
{
  "status": "dry_run",
  "wouldExecute": {
    "endpoint": "POST https://slack.com/api/chat.postMessage",
    "resolvedParams": { "channel": "C012345", "text": "Wide announcement" }
  }
}

Day-one API affordance even if the approval UI lands later. Phase 2: configure per credential — "Postgres-production always requires approval between dry-run and execute."

04Auth: extend the existing OAuth flow

n8n's MCP server is already an OAuth 2.1 provider: /mcp-oauth/register (DCR), /authorize, /token, /revoke, .well-known/oauth-authorization-server. Today's scopes: tool:listWorkflows, tool:getWorkflowDetails.

For hackathon we add three scopes and reuse everything else:

  • node:execute — call POST /executions/node
  • credential:read — list credentials by type/id (no secret material)
  • tool:search — call the three new MCP meta-tools

The existing consent screen stays. We update its scope-display strings and pre-register an n8n-cli OAuth client. Day-one "Connect to n8n" works for Claude Desktop and the CLI.

Out of scope for hackathon: polished Connected Apps page, per-token credential restriction UI, fancy revocation. The OAuth scaffolding handles revocation already.

05CLI: extend the existing n8n command

n8n's CLI lives at packages/cli/src/commands/ using @Command(...) decorators from @n8n/decorators. Existing commands (start, worker, execute) inherit BaseCommand, which boots the whole server. Too heavy for client commands talking to a remote instance — we add a lighter base.

packages/cli/src/
├── cli-client/                          # new
│   ├── config.ts                        # ~/.n8n/config.json read/write
│   ├── http-client.ts                   # axios wrapper, injects token
│   └── oauth-flow.ts                    # code-grant with localhost callback
└── commands/
    ├── remote-base-command.ts           # new — no DB init
    ├── auth/
    │   ├── login.ts                     # n8n auth login --instance <url>
    │   └── logout.ts                    # n8n auth logout
    ├── tools/
    │   └── search.ts                    # n8n tools search "<query>"
    └── exec/
        ├── run.ts                       # n8n exec <toolId> --cred <id> ...
        └── inspect.ts                   # n8n exec inspect <executionId>

Login flow (standard CLI OAuth pattern, like gh auth login)

  1. n8n auth login --instance https://n8n.company.io
  2. CLI starts a localhost listener on a random port.
  3. CLI opens the browser to /mcp-oauth/authorize?client_id=n8n-cli&redirect_uri=http://127.0.0.1:<port>&scope=node:execute+credential:read+tool:search.
  4. User clicks Authorize.
  5. n8n redirects to the localhost listener with ?code=....
  6. CLI exchanges the code at /mcp-oauth/token.
  7. CLI saves { instanceUrl, accessToken, refreshToken, expiresAt } to ~/.n8n/config.json (mode 0600).

Subsequent commands read that file and pass Authorization: Bearer <token>. Refresh handled transparently.

06SDK: greenfield @n8n/sdk with node types built-in

No existing external SDK. @n8n/workflow-sdk exists internally but is the wrong shape — it builds workflow JSON for the engine to execute later, not runtime method calls. Greenfield, reusing only the type-generation utilities.

packages/@n8n/sdk/
├── src/
│   ├── client.ts               # createClient({ baseUrl, token })
│   ├── proxy.ts                # n8n.slack.sendMessage(...) → POST /executions/node
│   ├── auth.ts                 # token storage + refresh
│   ├── pull.ts                 # CLI: refresh types from a specific instance
│   └── types/
│       ├── index.d.ts          # generated, ships with the package
│       └── nodes/              # one .d.ts per (node, version)
├── scripts/
│   └── generate-types.ts       # build-time codegen from nodes-base
└── package.json

Types built-in

At SDK build time, a script reads packages/nodes-base/dist/types/nodes.json (the existing catalog the editor and AI assistant already use) and emits per-node .d.ts files. They ship in the published npm package. After npm install @n8n/sdk, IDE autocomplete works for every action node in the standard catalog.

import { createClient } from '@n8n/sdk';

const n8n = createClient({
  baseUrl: 'https://n8n.company.io',
  token: process.env.N8N_TOKEN,  // OAuth access token from the CLI flow
});

// Full IDE autocomplete; types generated from the standard nodes-base catalog
await n8n.slack.sendMessage({
  credentialId: 'cred_3a8f',
  channel: '#engineering',
  text: 'Build is green',
});

Custom nodes via npx @n8n/sdk pull

Users with custom nodes run npx @n8n/sdk pull --instance https://n8n.company.io. The CLI fetches /types/nodes.json from that instance and writes overrides into node_modules/@n8n/sdk/.overrides/. Optional — most users don't need it.

Runtime is a thin Proxy

No client-side schema validation in v1; the server owns truth. The SDK assembles { nodeType, operation, credentialId, parameters } and dispatches to POST /executions/node. Errors surface as typed exceptions (N8nValidationError, N8nCredentialError, N8nExecutionError).

Edge cases (90/10)

  • resourceLocator — accept ID strings only at hackathon.
  • loadOptions (Slack channels, Notion DBs) — companion list operations.
  • resourceMapper — typed as Record<string, unknown>; runtime validates.

For operations we don't model, the HTTP Request node is in the catalog and accepts a credentialId — escape hatch for custom API calls without exposing secrets.

09What we don't know yet

displayOptions strategy
Every node has conditional field visibility. Two paths: encode rules in JSON Schema (complex), or expose all fields and document constraints in the tool description (simple, leaks complexity to the LLM). Prototype both, measure.
Utility nodes
Set, Code, IF, Function, Merge — they're transformations, not service actions. Default: exclude.
MCP server process model
NodeCatalogService lives in the CLI. If MCP server stays in-process, direct call. If they ever split, we expose the search via a backend endpoint. Not blocking — decide at implementation time.

Resolved by research: @n8n/workflow-sdk reusability — wrong shape, SDK greenfield.

10Risks (hackathon-relevant)

Tool-count problem ux

Naive registration of all node operations breaks LLM tool selection. Mitigated by the search-based meta-tool pattern (see live demo above).

Edge-case node properties technical

resourceLocator, loadOptions, resourceMapper don't map cleanly to JSON Schema or static TS. Mitigated by ID-only inputs at hackathon + the HTTP Request escape hatch in the SDK.
